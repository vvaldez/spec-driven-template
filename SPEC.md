# SPEC.md

## Vision and Goals
Describe, in a few paragraphs, what you are trying to build and why. Focus on the problem space and desired outcomes, not implementation details.

## Personas / Users
- Who are the primary users?
- What are their goals, constraints, and contexts?

## In-Scope vs Out-of-Scope
- **In scope**: What this project will address.
- **Out of scope**: Concise list of things you explicitly will not do (at least for now).

## High-Level Feature List
List the main features or capabilities you expect this project to have. Keep this at a summary level; details go in the next section.

## Feature Specifications
For each feature, use a consistent mini-template, for example:

### Feature: [Name]
- **Problem**: What issue this feature solves.
- **Proposed Solution**: How the feature addresses the problem at a high level.
- **User Flow**: Short description of how a user interacts with this feature.
- **Acceptance Criteria**: Clear, testable statements that define “done” for this feature.

Repeat this subsection for each significant feature.

## Non-Functional Considerations
Capture important qualities that affect design, such as:
- Performance and responsiveness.
- Reliability and availability.
- Accessibility and inclusivity.
- Observability and maintainability.

## Links to Related Docs
Once you adopt advanced docs, link them here (for example):
- Architecture overview: `advanced/ARCHITECTURE.md`
- Data model: `advanced/DATA_MODEL.md`
- Decisions: `advanced/DECISIONS.md`
- Risks: `advanced/RISKS.md`
